# 관찰성 목표 정리와 Telemetry Layer 첫 구현

## 1. 배경: 관찰성은 처음부터 MVP의 핵심이었다

LaunchPilot은 처음부터 관찰성을 MVP의 중요한 요소로 보고 시작했습니다.

우리가 만드는 것은 단순한 API 서버가 아니라, 사용자의 요청을 이해하고, 캠페인 데이터를 확인하고, 신호를 찾고, 가설을 세우고, 실험 계획까지 제안하는 에이전트입니다.

이런 시스템에서는 "응답이 왔다"만으로 충분하지 않습니다.

우리는 에이전트가 어떤 근거를 봤는지, 어떤 단계를 거쳐 답을 만들었는지, 결과가 믿을 만한지, 그리고 그 과정이 너무 느리거나 비싸지는 않은지를 함께 봐야 합니다.

그래서 Arize/Phoenix 기반 관찰성이 MVP에 포함되었습니다.

이제 초기 구현이 들어간 상태에서, 팀이 같은 기준으로 관찰성을 바라볼 수 있도록 목표와 코드 구조를 한 번 정리하려고 합니다.

## 2. 우리가 관찰하려는 두 가지

관찰성 목표는 크게 두 축으로 나눠보려고 합니다.

### [1] 답변 품질

여기서 보고 싶은 것은 에이전트가 "그럴듯한 말"을 했는지가 아니라, 실제로 믿을 만한 답을 만들었는지입니다.

예를 들면 이런 질문입니다.

- 사용자의 요청을 제대로 이해했는가?
- 필요한 캠페인 데이터를 실제로 확인했는가?
- 분석 결과가 근거 있는 신호인가?
- 가설이 너무 단정적이거나 근거 없이 과장되어 있지는 않은가?
- 실험 계획이 실제로 실행 가능하고 측정 가능한가?
- 최종 결과가 reviewer 기준을 통과했는가?

### [2] 실행 효율

좋은 답을 만들더라도 매번 너무 오래 걸리거나, LLM을 과도하게 호출하거나, 같은 데이터를 반복해서 조회하면 제품으로 운영하기 어렵습니다.

그래서 이런 것도 함께 보고 싶습니다.

- 한 번의 요청을 처리하는 데 얼마나 걸렸는가?
- LLM을 몇 번 호출했는가?
- 외부 데이터 조회를 몇 번 했는가?
- 중간에 실패하거나 재시도한 적이 있는가?
- 캐시를 잘 활용했는가?
- 같은 요청이나 같은 데이터 조회를 불필요하게 반복하고 있지는 않은가?
- 좋은 답변을 만들기 위해 너무 많은 비용을 쓰고 있지는 않은가?

이 두 축을 함께 보는 이유는 간단합니다.

답변 품질만 보면 "좋긴 한데 너무 느리고 비싼 에이전트"가 될 수 있습니다.

실행 효율만 보면 "빠르긴 한데 믿기 어려운 에이전트"가 될 수 있습니다.

우리가 원하는 것은 둘 중 하나가 아니라, **믿을 만한 답을 적절한 시간과 비용 안에서 안정적으로 만드는 에이전트**입니다.

## 3. 현재 이미 되어 있는 것

현재 코드에는 관찰성의 기초가 이미 들어가 있습니다.

- Google ADK 실행 흐름을 OpenInference 방식으로 추적할 수 있습니다.
- Phoenix/Arize로 trace를 보낼 수 있는 설정이 있습니다.
- 한 턴 처리, 파이프라인 실행, 데이터 조회, reviewer gate 같은 주요 구간이 span으로 남습니다.
- 실험 계획이 최소 기준을 만족하는지 확인하는 deterministic reviewer가 있습니다.
- 별도 eval 흐름에서는 결과 품질을 점수화하는 LLM judge도 준비되어 있습니다.

즉 관찰성은 나중에 급히 붙인 부가 기능이 아니라, 초기 설계부터 포함된 중요한 축입니다.

## 4. 현재 구조에서 정리하고 싶은 점

관찰성 helper는 현재 `app/infra_observability`, `app/telemetry`, `app/tracing`, `app/phoenix_export`로 나뉘어 있습니다. `infra_observability`는 service-level logs/metrics/traces와 Alloy export를 맡고, `telemetry`와 `tracing`은 도메인 span/metadata를 맡으며, `phoenix_export`는 Phoenix/OpenInference export를 맡습니다. 다만 실제 사용 방식은 여러 에이전트 로직 사이에 직접 들어가 있습니다.

예를 들어 workflow, router, phase runner, evidence tool, committer 같은 코드에서 직접 span을 열거나 metadata를 붙이고 있습니다.

지금은 괜찮습니다. 아직 규모가 크지 않고, 어떤 일이 일어나는지도 비교적 잘 보입니다.

하지만 앞으로 다음 항목을 더 붙이기 시작하면 관찰성 코드가 더 많이 퍼질 수 있습니다.

- 품질 점수
- LLM 호출 수
- tool 호출 수
- token 사용량
- 예상 비용
- cache hit/miss
- 중복 요청 감지 여부
- 같은 evidence query 반복 횟수
- 같은 phase 재실행 횟수
- trace id 연결
- Phoenix reflection 결과
- evaluator 결과

이렇게 되면 나중에 몇 가지 문제가 생길 수 있습니다.

- 어떤 지표가 어디서 기록되는지 찾기 어려워질 수 있습니다.
- 같은 값을 여러 곳에서 다른 이름으로 남길 수 있습니다.
- 민감한 입력이나 출력이 trace에 섞이지 않도록 관리하기 어려워질 수 있습니다.
- Phoenix/Arize 외의 다른 관찰성 도구를 붙이거나 바꿀 때 수정 범위가 커질 수 있습니다.
- 에이전트 로직 자체가 점점 읽기 어려워질 수 있습니다.

그래서 지금 필요한 것은 "관찰성을 새로 붙이자"가 아니라, **이미 들어간 관찰성을 제품 목표와 코드 구조에 맞게 정돈하자**에 가깝습니다.

## 5. 제안하는 방향

앞으로는 에이전트 로직이 OpenTelemetry나 OpenInference의 세부 구현을 직접 알지 않도록 하고 싶습니다.

에이전트 로직은 "무슨 일이 일어났는지"만 말하고, 관찰성 모듈이 "그 일을 어떤 표준과 형식으로 기록할지"를 책임지는 구조가 좋겠습니다.

예를 들면 에이전트 코드에서는 이렇게만 표현합니다.

```python
telemetry.turn_started(...)
telemetry.intent_interpreted(...)
telemetry.evidence_checked(...)
telemetry.guardrail_checked(...)
telemetry.quality_scored(...)
telemetry.cost_recorded(...)
telemetry.duplicate_work_detected(...)
telemetry.turn_finished(...)
```

그리고 `telemetry` 모듈 안에서 다음을 처리합니다.

- 이 이벤트를 어떤 span으로 남길지
- OpenInference 기준으로 어떤 kind를 붙일지
- OpenTelemetry metadata에는 어떤 값을 넣을지
- 어떤 입력/출력은 저장하지 말아야 하는지
- trace id를 어디에 연결할지
- Phoenix/Arize에서 보기 좋은 이름은 무엇인지

쉽게 말하면, 도메인 코드는 제품 언어로 말하고, telemetry 모듈이 관찰성 표준 언어로 번역하는 구조입니다.

## 6. 이번 PR에서 하고 싶은 일

이번 PR에서 모든 것을 한 번에 바꾸자는 뜻은 아닙니다. 우선 팀이 같은 그림을 볼 수 있도록 기준을 세우고, 아주 얇은 telemetry layer를 먼저 추가하는 것이 목적입니다.

이번 범위는 다음 정도로 생각합니다.

- 관찰성 목표를 `답변 품질`과 `실행 효율` 두 축으로 정리
- 현재 어떤 관찰성 지점이 있는지 정리
- 지금 빠져 있거나 약한 지점 목록화
- `telemetry` layer가 어떤 역할을 가져야 하는지 초안 제안
- 기존 `tracing.*` 직접 호출 일부를 `telemetry.*` 호출로 옮겨 첫 적용 예시 만들기
- 앞으로 어떤 순서로 개선할지 논의할 수 있는 기반 마련

## 7. 이번 PR에서 하지 않는 일

다음은 이번 PR에서 바로 하지는 않으려고 합니다.

- 기존 tracing 코드를 전부 한 번에 옮기기
- live runtime에 LLM judge를 바로 강제 연결하기
- Phoenix/Arize 대시보드를 완성하기
- token 비용 산정 모델을 확정하기
- 관찰성 도구를 바꾸기
- 모든 기존 tracing 호출을 한 번에 telemetry로 이관하기

지금 단계에서는 큰 변경보다, 팀이 같은 기준을 공유하는 것이 더 중요하다고 봅니다.

## 8. 팀과 논의하고 싶은 질문

이 PR에서 팀과 맞추고 싶은 질문은 세 가지입니다.

1. LaunchPilot 관찰성 목표를 **답변 품질**과 **실행 효율** 두 축으로 나누는 데 동의하는가?
2. 품질 점수화를 실제 사용자 요청 처리 흐름에도 붙일 것인가, 아니면 당분간 별도 eval 흐름에서만 볼 것인가?
3. 앞으로 새 관찰성 지표는 직접 `tracing.*`를 호출하지 않고, 의미 중심의 `telemetry` 모듈을 통해 기록하는 방향에 동의하는가?

## 9. 정리

LaunchPilot에서 관찰성은 부가 기능이 아니라, 에이전트가 신뢰할 만한 판단을 하고 있는지 확인하기 위한 핵심 장치입니다.

이번 정리는 "trace를 더 많이 찍자"가 아니라, **무엇을 보기 위해 trace를 찍는지**를 분명히 하려는 작업입니다.

우리가 보고 싶은 것은 결국 두 가지입니다.

- 에이전트가 믿을 만한 답을 만들었는가?
- 그 답을 합리적인 시간과 비용 안에서 만들었는가?

이 기준을 먼저 맞춰두면, 앞으로 품질 평가, 비용 추적, trace id 연결, Phoenix reflection 같은 기능을 붙일 때 코드도 더 깔끔하게 유지할 수 있을 것입니다.
