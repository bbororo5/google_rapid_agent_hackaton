# 엔지니어링 규율 결정

다언어 병렬 개발의 경계 관리, 데모 안정성에 관한 두 결정.

---

## ADR-0011 · Contract-first (asyncapi/openapi/schema, extra=forbid)

**상태:** 채택

**맥락:**
시스템은 세 언어, 세 컨테이너(Next.js, Java, Python)로 나뉘어 병렬 개발된다. 경계는 여섯 군데다(프론트↔Java, Java↔Python, Java↔Elastic, Python↔검색 MCP, 워커↔출력, 관측성). 한쪽이 필드를 변경하면 드리프트가 발생해 통합 단계에서 드러난다. 해커톤 일정상 통합 디버깅에 쓸 시간이 없다.

**결정:**
모든 경계를 `contracts/` 아래 기계 판독 가능한 계약으로 고정한다(REST는 OpenAPI, WebSocket은 AsyncAPI, 문서·도구·출력은 각 JSON 스키마). Python은 계약을 Pydantic 모델로 옮기되 미정의 필드를 즉시 거부하도록 `extra=forbid`와 pattern 제약을 적용한다. 계약 예시는 적합성 테스트로 강제한다.

**결과:**
세 팀이 계약만 보고 병렬 개발한다. 드리프트가 통합 단계가 아닌 작성 시점에 잡힌다. 계약 예시가 테스트가 된다. 초기 개발 속도를 포기한다(경계 선합의 비용). 계약 변경 시 스키마·타입·테스트를 함께 수정해야 한다. `extra=forbid`이므로 미정의 필드는 즉시 거부된다.

**대안:**
- 느슨한 JSON + 런타임 신뢰: 드리프트가 통합 막바지에 드러나고 다언어 디버깅 비용이 크다.
- 코드에서 스키마 자동 생성: 세 언어 중 단일 진실원을 합의하기 어렵고 경계가 비대칭이 된다.

## ADR-0012 · CSV import + 결정적 fallback (실시간 SNS API 미연동)

**상태:** 채택

**맥락:**
해커톤 제출의 성패는 3분 라이브 데모의 안정성이다. Instagram/TikTok/X 공식 API는 인증·레이트리밋·승인 절차가 무겁고 데모 중 실패 위험이 크다. Gemini 호출 지연·실패 시 데모가 중단된다. 평가 대상은 실시간 데이터 수집이 아닌 신호→실험 사고의 자동화다.

**결정:**
실시간 SNS 공식 API를 연동하지 않는다. CSV import로 성과 데이터를 받고 Java가 스트리밍 파싱 후 Elastic에 즉시 인덱싱한다. 외부 의존이 없을 때를 위해 결정적 fallback을 둔다. `GEMINI_API_KEY`/Vertex 설정이 없으면 stub 에이전트(오프라인 결정론)가, `ELASTIC_MCP_URL`이 없으면 seed 데이터가 동작한다. 외부 키 없이도 전체 파이프라인이 오프라인으로 동작한다.

**결과:**
안정적인 3분 데모가 가능하다. 네트워크·키 없이 오프라인 전체 동작과 외부 의존 없는 테스트가 가능하다. 모델 지연에도 흐름이 유지된다. 실시간성을 포기한다. 데모는 준비된 데이터셋에 한정되고 실제 SNS API 경로는 미검증이다. stub 출력은 결정론이므로 실제 모델 품질을 대표하지 않는다.

**대안:**
- 실시간 공식 API 연동: 인증·레이트리밋이 무겁고 데모 실패 위험이 크며 평가 핵심과 무관하다.
- fallback 없이 실 키 강제: 키 부재·지연 시 데모와 테스트가 중단된다.

---

관련 계약: `contracts/`의 여섯 폴더, `apps/agent/app/contracts/schemas.py`(Pydantic 계약). 2모드 폴백 구현: `apps/agent`.
