# 4. 어떻게 안정적으로 만드는가

> 상위 문서: [아키텍처 개요](../overview.md) · 구조 그림: [C4](../launchpilot-c4.md) §6

좋은 아키텍처도 짓는 과정이 무너지면 의미가 없다. 마지막 두 결정은 화려하지 않지만, 세 개 팀이 동시에 짜는 코드가 통합 단계에서 터지지 않게 하고, 3분짜리 라이브 데모가 무대 위에서 멈추지 않게 한다.

## 계약을 먼저, 코드를 나중

이 시스템은 세 개의 언어와 세 개의 컨테이너로 나뉜다. Next.js 프론트엔드, Java 백엔드, Python 에이전트 서비스가 동시에, 병렬로 개발된다. 그리고 이들이 만나는 경계가 여섯 군데다 — 프론트와 Java, Java와 Python, Java와 Elastic, Python과 검색 MCP, 워커와 그 출력, 그리고 관측성.

경계가 많은 병렬 개발의 고질병은 조용한 드리프트다. 한쪽이 필드 하나를 슬쩍 바꾸면, 그 어긋남은 통합하는 막바지에 가서야 터진다. 그리고 세 개 언어에 걸친 통합 디버깅은 시간을 삼킨다 — 해커톤 일정에는 없는 시간이다.

그래서 코드보다 계약을 먼저 정했다. 여섯 경계 전부를 기계가 읽을 수 있는 계약으로 못 박는다 — REST는 OpenAPI, WebSocket은 AsyncAPI, 문서와 도구와 출력은 각각의 JSON 스키마로. Python은 이 계약을 그대로 Pydantic 모델로 옮기되, 정의되지 않은 필드는 즉시 거부하도록(`extra=forbid`) 강제한다. 드리프트가 통합 막판이 아니라 코드를 쓰는 그 순간에 잡히게 만드는 그물이다. 계약에 담긴 예시는 그대로 적합성 테스트가 된다.

대가는 초기 속도다. 무엇이든 짜기 전에 경계를 먼저 합의하는 선행 비용이 든다. 계약을 바꿀 때는 스키마, 타입, 테스트를 함께 고쳐야 한다. 그러나 이 규율이 세 팀에게 "계약만 보면 된다"는 독립성을 주고, 통합의 밤을 평범한 오후로 바꾼다.

> 버린 길: 느슨한 JSON에 런타임 신뢰(드리프트가 막판에 터지고 세 언어 디버깅 지옥), 코드에서 스키마 자동 생성(세 언어 중 무엇을 단일 진실로 삼을지 합의가 안 되고 경계가 비대칭이 됨).

## 실시간 대신 안정적인 3분

해커톤 제출의 성패는 무대 위 3분에 달려 있다. 그런데 Instagram·TikTok·X의 공식 API는 인증, 레이트리밋, 승인 절차가 무겁고, 데모 도중 실패할 위험이 크다. 게다가 평가받는 것은 실시간으로 데이터를 긁어오는 능력이 아니라, 데이터를 신호에서 실험으로 바꾸는 사고의 자동화다.

그래서 실시간 SNS API를 일부러 연동하지 않았다. 대신 CSV 업로드로 성과 데이터를 받는다. Java가 이를 스트리밍으로 파싱해 Elastic에 즉시 색인한다. 여기에 더해, 외부 의존이 없을 때를 대비한 결정적 대체 경로(fallback)를 깔았다. Gemini 키가 없으면 오프라인 결정론으로 동작하는 스텁 에이전트가 돌고, Elastic 연결이 없으면 시드 데이터가 그 자리를 채운다. 그 결과 외부 키 하나 없이도 전체 파이프라인이 오프라인에서 끝까지 동작한다.

포기하는 것은 실시간성이다 — 라이브 SNS 데이터를 자동으로 수집하는 능력은 범위 밖이다. 그래서 한계도 분명하다. 데모는 준비된 데이터셋 위에서만 돌고, 실제 SNS API 경로는 검증되지 않았으며, 스텁의 출력은 결정론이라 실제 Gemini의 품질을 대표하지 않는다. 그러나 이 선택이 주는 것은, 네트워크가 흔들리고 모델이 지연돼도 멈추지 않는 데모와, 외부 의존 없이 돌아가는 테스트다. 무대 위에서는 안정성이 화려함을 이긴다.

> 버린 길: 실시간 공식 API 연동(인증·레이트리밋이 무겁고 데모 실패 위험이 크며 평가 핵심과 무관), 대체 경로 없이 실 키 강제(키가 없거나 지연되면 데모와 테스트가 멈춤).
