# 투명성 · 관측성 결정

추론 과정의 실시간 전달, 노출 범위, 과거 실행 성찰에 관한 세 결정.

---

## ADR-0008 · 영속 WebSocket 타임라인 + 재접속 리플레이 (폴링 미사용)

**상태:** 채택

**맥락:**
제품 차별점은 에이전트 추론 과정의 실시간 관찰(glass-box)이다. 폴링은 단계별 진행을 라이브로 전달하지 못한다. 라이브 스트림은 연결이 끊기면 그 사이 이벤트를 잃으며, 그중에는 승인 게이트 신호와 브리프 ID처럼 유실되면 안 되는 것이 포함된다. 새 세션·재접속에서 타임라인을 복원할 수 있어야 한다.

**결정:**
WebSocket을 주 채널로, REST를 시작/승인/복구 fallback으로 사용하는 이중 채널을 채택한다. WS 타임라인은 영속적이며 모든 이벤트에 순번(sequence)을 부여한다. 끊김 후 `connection.resume(last_sequence)`로 누락분만 리플레이하고, 증분 불가 시 `connection.full_sync`로 전체를 재생한다. 승인 결과는 `approval.committed`(brief_id 포함)로 WS로 반환한다. REST `GET`은 이벤트 히스토리가 아닌 현재 상태의 coarse 스냅샷이다. 계약은 `01/02 asyncapi.yaml`로 강제한다.

**결과:**
라이브 glass-box 타임라인을 제공한다. 끊김에도 이벤트가 무손실로 복구된다. 새 세션에서 타임라인을 복원한다. 폴링의 단순함을 포기한다. 이벤트 순번 부여·저장과 타임라인 영속화 비용이 발생한다. WS 연결 수명 관리가 필요하다. 서버 스레드 고갈 방지를 위해 별도 TaskExecutor 풀에서 실행한다.

**대안:**
- REST 폴링 전용: 라이브 느낌이 없고 이벤트가 유실·중복된다.
- 영속성 없는 WebSocket: 끊기면 그 사이 이벤트(승인 신호 포함)가 영구 손실된다.

## ADR-0009 · Glass-box는 정규화 이벤트만 전송 (원시 출력 비전송)

**상태:** 채택

**맥락:**
추론을 실시간 전달하되 Gemini 원시 출력에 포함된 chain-of-thought, `thoughtSignature`, 함수 호출 전송 메시지, 원시 MCP 메시지는 노출할 수 없다. 그대로 전달하면 내부 신호가 노출되고, 화면이 노이즈로 덮이며, 프론트엔드가 특정 제공자의 출력 포맷에 결합된다.

**결정:**
glass-box 스트리밍은 정규화된 워크플로우 이벤트만 전송한다. Python은 원시 청크·사고 흔적·원시 MCP 메시지를 전송하지 않고, 사람이 봐도 되는 형태(`observation`, `signal`, `hypothesis`, `plan`)로 정규화한 이벤트만 스트리밍한다. Java는 이를 받아 glass-box 정책을 적용한 뒤 프론트로 릴레이한다.

**결과:**
안전하고 가독성 있는 공개 추론을 제공한다. 프론트엔드가 제공자 포맷과 분리된다. 완전한 원시 투명성을 포기한다. 워커 출력을 정규화 이벤트로 변환하는 오버헤드가 발생한다. 원시 진단이 필요하면 OpenInference 트레이스(L4, ADR-0010)에서 확인한다.

**대안:**
- 원시 Gemini 스트림 전달: 내부 신호 노출, 노이즈, 프론트-제공자 결합, 보안 위험.
- 최종 결과만 노출: glass-box 차별점이 사라진다.

## ADR-0010 · Phoenix/Arize L4 자가 성찰

**상태:** 채택

**맥락:**
에이전트가 같은 실수를 반복하는 것은 흔한 실패다. 단순 로깅은 기록을 남기지만 과거 실패 패턴을 다음 추론에 반영하지 못한다. 추론 경로를 트레이스로 추적할 수 있어야 에이전트성이 증명된다.

**결정:**
기억의 네 계층(ADR-0002) 중 성찰 계층(L4)을 Phoenix/Arize에 둔다. 모든 LLM 호출, 도구 호출, 검수자 판정을 OpenTelemetry/OpenInference 스팬으로 실시간 계측해 송신한다. 검수자(ADR-0006)는 검증 전에 Phoenix MCP(`get_traces`/`get_evaluations`)로 과거 낮은 평가 패턴을 조회해 검증 기준과 추론을 보정한다.

**결과:**
과거 실패를 학습해 반복을 줄인다. 추론 경로가 트레이스로 추적 가능하다. 순수 추가 기능이므로 포기하는 것은 없다(연동·비용 부담은 있다). 성찰은 검증 기준 보정용이며 사용자에게 제시할 근거(evidence)가 아니다. 트레이스 수준 관측의 진실 공급원은 OpenInference 스팬이고, 화면의 도구 호출 로그는 UI 요약용이다.

**대안:**
- 관측성 없이 단순 로그: 과거 실패 재학습과 추론 추적이 불가능하다.
- 자체 트레이스 저장소 구축: 해커톤 범위를 초과하고 OpenInference 표준의 이점을 잃는다.
