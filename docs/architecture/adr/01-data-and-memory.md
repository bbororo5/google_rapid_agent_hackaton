# 1. 데이터를 어디에, 어떻게 두는가

> 상위 문서: [아키텍처 개요](../overview.md) · 구조 그림: [C4](../launchpilot-c4.md)

데이터 설계는 보통 가장 늦게 정해지지만, LaunchPilot에서는 가장 먼저 무너지고 가장 먼저 다시 세워진 부분이다. 세 가지 결정이 여기서 맞물린다. 무엇을 저장소로 쓸 것인가, 에이전트의 기억을 어떻게 다룰 것인가, 그리고 사람이 승인하기 전의 데이터는 어디에 둘 것인가.

## 저장소를 하나로 합치다

제품의 가장 중요한 동작은 근거를 모으는 일이다. 전략가 워커가 "저장률이 평소의 2.8배로 튀었다"는 정량 신호에 "팬들이 연습실 클립에 강하게 반응한다"는 팀 메모를 엮어 가설을 세우려면, 정형 지표 집계와 비정형 텍스트의 시맨틱 검색이 같은 추론 흐름 안에서 일어나야 한다.

관계형 DB는 이 조합에 약하다. 검색엔진을 따로 붙이는 길도 있었지만, 그러면 같은 데이터를 두 저장소에 넣고 끊임없이 동기화해야 한다. 게다가 데모에서는 CSV를 올린 직후 곧바로 검색이 돼야 하는데, 이 실시간성을 두 시스템에 걸쳐 보장하기는 까다롭다.

그래서 별도의 애플리케이션 DB를 두지 않기로 했다. Elastic Cloud Serverless 하나가 유일한 저장소이자 근거 엔진이 된다. 모든 도메인 데이터 — 콘텐츠 성과, 팔로워 시계열, 캠페인, 캘린더, 팀 메모, 승인된 과거 브리프 — 가 Elastic 인덱스로 들어간다. Java 백엔드는 관계형 DB 없이 Elastic에 곧바로 인덱싱하는 얇은 게이트웨이로 줄어든다. CSV가 들어오면 `refresh=true`로 즉시 색인돼, 업로드와 동시에 검색 대상이 된다.

대가는 분명하다. 트랜잭션, 멀티 테이블 조인, 외래키 무결성, 그리고 흔한 update/delete 중심의 CRUD를 포기한다. 엔티티 사이의 관계는 DB가 아니라 애플리케이션 레벨에서 ID 참조로 표현하고, 그 참조가 깨지지 않았는지는 [검수자](02-the-agents.md)가 검증한다. 단일 저장소가 준 단순함의 값으로 치르는 비용이다.

> 버린 길: 관계형 DB + 별도 검색 인덱스(이중 적재·동기화), Elastic + 얇은 메타 DB 병행(단일 저장소의 단순함을 도로 깨뜨림).

## 기억을 한 통으로 보지 않는다

"에이전트의 기억"을 단일한 무언가로 상상하면 설계가 곧 꼬인다. 워커들이 한 번의 실행 안에서 주고받는 임시 상태, 사용자와의 대화 타임라인, 영구히 쌓이는 지식, 과거 실행의 트레이스 — 이 넷은 수명도, 주인도, 접근 권한도 전부 다르기 때문이다. 이걸 "단기 기억" 같은 한 단어로 뭉뚱그리면 누가 쓰고 누가 읽는지, 언제 사라지는지가 흐려진다.

그래서 기억을 네 층으로 갈라 각각의 수명과 소유자를 못 박았다.

**런 메모리**는 Python 프로세스 안에만 사는 인메모리 상태다. 한 번의 실행 동안 신호 → 가설 → 실험이 단계별로 쌓이고, 실행이 끝나면 휘발한다. 워커들이 결과를 주고받고 백트래킹 때 앞 단계를 재사용하는 작업대일 뿐, 영구 저장이 아니다. 프로세스가 죽으면 실행은 실패로 처리되고 다시 시작한다.

**세션**은 Java가 소유한다. 메시지 타임라인, 승인 이력, 순번(sequence) 기반 리플레이가 여기 산다. 런 메모리와 절대 섞지 않는다.

**지식 층(L3)**이 앞서 말한 Elastic이다. 영구 저장소이며, 쓰기는 Java만, 읽기는 Python이 MCP로만 한다.

**성찰 층(L4)**은 Phoenix/Arize다. 한 번의 실행을 넘어 누적되는 트레이스로, [추론을 보여주고 추적하는](03-transparency.md) 이야기로 이어진다.

이 분리에는 깨면 안 되는 규칙이 따라붙는다. Elastic에 쓰는 주체는 Java뿐이고, 그것도 한 번의 실행에 딱 두 번 — 데이터 임포트와 사용자 승인 — 만이다. 워커 중에서는 분석가와 전략가만 Elastic을 읽고, 작가와 검수자는 읽지 않는다. 층의 경계를 코드와 계약으로 강제하는 규율이 이 설계의 값이다.

> 버린 길: 세션·Elastic 메타·트레이스를 하나의 "단기 기억"으로 추상화(소유와 수명이 뒤엉킴), 에이전트가 Elastic에 직접 쓰기(쓰기 권한이 흩어져 불변성과 감사성이 무너짐).

## 승인 전에는 아무것도 저장하지 않는다

에이전트가 만든 실험안은 제안일 뿐이다. 그것이 캘린더와 브리프에 반영되려면 사람의 손을 거쳐야 한다. 만약 Python 에이전트가 결과를 곧바로 저장한다면, 사람이 검토하고 고치기도 전에 데이터가 확정돼 버린다 — 자율 실행이 돼버리는 것이다. 또한 승인 전 후보를 어딘가에 영구 저장하면, 채택되지 않은 초안이 저장소에 쌓여 앞서 정한 append-only 원칙과 충돌한다.

그래서 경계를 분명히 그었다. 프론트엔드는 상태를 영속화하지 않는다(stateless). 승인 전 후보 실험안은 오직 화면의 메모리(React State)에만 존재하며, 사용자의 편집도 초안 사본에만 적용되고 원본은 불변으로 남는다. 그리고 **승인이라는 관문 자체를 Java가 소유한다.** Python은 "승인 대기" 상태 신호까지만 책임지고, 실제 승인 처리에는 관여하지 않는다. 사람이 승인 버튼을 누르는 순간에만 Java가 브리프 한 건과 캘린더 이벤트 여러 건을 불변으로 적재한다.

이 결정이 포기하는 것은 에이전트의 완전한 자율성이다. 시스템은 스스로 게시하지 않고, 스스로 확정하지 않는다. 그 대가로 얻는 것은 사람이 항상 마지막 결정권을 쥔다는 안전성과, 저장소에는 승인된 것만 남는 깨끗함이다. 다만 이 선택에는 작은 그늘이 있다 — 승인 전 후보는 휘발성이라, 편집하던 초안은 브라우저를 새로고침하면 사라진다.

> 버린 길: Python이 결과를 직접 저장(human-in-the-loop가 깨지고 미승인 데이터가 쌓임), 후보를 서버 임시 저장소에 보관(별도 저장소와 정리 로직이 필요해지고 단일 저장소 원칙을 위배).
