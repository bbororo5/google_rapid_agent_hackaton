# 데이터 · 기억 결정

데이터의 저장 위치, 에이전트 기억의 계층, 승인 전 데이터의 처리에 관한 세 결정.

---

## ADR-0001 · Elastic Cloud Serverless를 유일 데이터 저장소로 사용

**상태:** 채택

**맥락:**
가설 수립에는 정형 지표 집계(예: 저장률의 베이스라인 대비 배수)와 비정형 텍스트의 시맨틱 검색(예: 팀 메모)이 같은 추론 흐름 안에서 함께 필요하다. 관계형 DB는 이 하이브리드 검색을 단독으로 지원하지 못한다. 검색엔진을 별도로 두면 동일 데이터를 두 저장소에 적재하고 동기화해야 한다. 데모 요건상 CSV 업로드 직후 즉시 검색이 가능해야 한다. 해커톤 트랙은 Elastic이다.

**결정:**
별도 애플리케이션 DB(Postgres/SQLite)를 두지 않는다. Elastic Cloud Serverless 하나가 유일한 데이터 저장소이자 근거 엔진이다. 모든 도메인 엔티티(`content_posts`, `follower_logs`, `campaigns`, `calendar_events`, `team_notes`, `growth_briefs`)는 Elastic 인덱스다. Java 백엔드는 관계형 DB 없이 Elastic에 `refresh=true`로 직접 인덱싱하는 게이트웨이다.

**결과:**
정형 집계와 비정형 검색을 단일 엔진에서 제공한다. 이중 저장소 동기화가 사라진다. CSV import 직후 즉시 검색이 가능하다. 트랜잭션, 멀티 테이블 조인, 외래키 무결성, update/delete 중심 CRUD를 포기한다. 엔티티 관계는 애플리케이션 레벨의 ID 참조로 표현하며, 참조 무결성은 검수자(ADR-0006)가 검증한다. 데이터는 append-only로 다룬다(ADR-0003).

**대안:**
- 관계형 DB + 별도 검색 인덱스: 이중 적재와 동기화 비용이 발생하고 하이브리드 검색이 일급이 아니다.
- Elastic + 얇은 메타 DB 병행: 단일 저장소의 단순함을 잃고 동기화 문제를 다시 도입한다.

## ADR-0002 · 에이전트 기억을 네 계층으로 분리

**상태:** 채택

**맥락:**
워커가 한 실행 안에서 주고받는 임시 상태, 사용자 대화 타임라인, 영구 지식, 과거 실행 트레이스는 수명·소유자·접근 권한이 모두 다르다. 이를 단일한 "단기 기억"으로 다루면 쓰기/읽기 주체와 소멸 시점이 모호해진다.

**결정:**
기억을 네 계층으로 분리하고 각 계층의 수명·소유자·접근을 고정한다.

| 계층 | 저장소 | 소유 | 수명 |
|---|---|---|---|
| 런 메모리 (L1/L2) | Shared Context Object (Python 인메모리, Pydantic) | Python | 실행 1회, 휘발 |
| 세션 | Java | Java | 메시지 타임라인·승인·순번 리플레이 |
| 지식 (L3) | Elastic | Java 쓰기 / Python 읽기전용(MCP) | 영구 |
| 성찰 (L4) | Arize/Phoenix | Python 트레이스 쓰기 | 실행 누적 |

불변 규칙: Elastic 쓰기는 Java만, 실행당 두 번(데이터 import, 사용자 승인)으로 제한한다. 워커 중 분석가·전략가만 Elastic을 읽고, 작가·검수자는 읽지 않는다. 런 메모리는 영구 저장이 아니다.

**결과:**
각 계층의 쓰기/읽기 주체와 소멸 시점이 명확해진다. 성찰 계층(L4)이 근거 데이터와 분리된다(ADR-0010). 런 메모리의 프리픽스 재사용으로 백트래킹이 효율적이다(ADR-0007). 계층 경계를 코드와 계약으로 강제하는 비용이 발생한다. 런 메모리는 프로세스 종료 시 소실되며, 이 경우 실행은 실패로 처리되고 재시작한다.

**대안:**
- 단일 "단기 기억" 추상화: 세션·Elastic 메타·트레이스를 뭉뚱그려 소유와 수명이 뒤엉킨다.
- 에이전트가 Elastic에 직접 쓰기: 쓰기 권한이 분산되어 불변성과 감사성이 무너진다.

## ADR-0003 · 승인 전 데이터 비저장, 승인 게이트는 Java 소유

**상태:** 채택

**맥락:**
에이전트가 만든 실험안은 사람의 검토·승인을 거쳐야 캘린더와 브리프에 반영된다. Python이 결과를 직접 저장하면 검토 전에 데이터가 확정되어 자율 실행이 된다. 승인 전 후보를 영구 저장하면 미채택 초안이 저장소에 쌓여 append-only 원칙(ADR-0001)과 충돌한다.

**결정:**
프론트엔드는 상태를 영속화하지 않는다(stateless). 승인 전 후보 실험안은 화면 메모리(React State)에만 존재한다. 사용자 편집은 초안 사본에만 적용하고 원본은 불변으로 둔다. 승인 게이트는 Java가 소유한다. Python은 `WAITING_FOR_APPROVAL` 상태 신호까지만 담당하고 승인을 처리하지 않는다. 승인 시에만 Java가 `growth_briefs` 1건과 `calendar_events` N건을 불변으로 적재한다.

**결과:**
사람이 항상 마지막 결정권을 가진다. 저장소에는 승인된 데이터만 남는다. 책임이 분리된다(Python은 추론, Java는 승인 라이프사이클). 에이전트의 자율 실행을 포기한다. 승인 전 후보는 휘발성이며 브라우저 새로고침 시 소실된다. `SUCCESS`는 Java 승인 라이프사이클 전용이고 Python은 방출하지 않는다.

**대안:**
- Python이 결과를 직접 저장: human-in-the-loop가 깨지고 미승인 데이터가 적재된다.
- 후보를 서버 임시 저장소에 보관: 별도 저장소와 정리 로직이 필요하고 단일 저장소 원칙에 위배된다.
