# 2. AI를 어떻게 굴리는가

> 상위 문서: [아키텍처 개요](../overview.md) · 구조 그림: [C4](../launchpilot-c4.md) §4

[개요](../overview.md)에서 본 재구성 — AI를 신탁이 아니라 팀으로 다루기로 한 전환 — 이 가장 직접적으로 드러나는 곳이 이 장이다. 네 개의 결정이 그 전환을 떠받친다. 에이전트를 어떻게 나눌 것인가, 무엇으로 굴릴 것인가, 결과를 누가 검증할 것인가, 그리고 검증이 실패했을 때 무엇을 다시 시킬 것인가.

## 한 명이 아니라 네 명

제품 루프 자체가 단계로 나뉘어 있다. 신호를 찾고, 원인을 가설화하고, 실험을 설계하는 일은 성격이 전혀 다른 작업이다. 사람 조직이라면 한 사람에게 다 시키지 않는다. 시스템도 그렇게 했다.

역할이 분리된 네 명의 워커가 일을 나눠 맡는다. **분석가**가 정량 신호를 탐지하고, **전략가**가 그 신호의 원인을 가설로 세우며, **작가**가 가설을 검증 가능한 다음 주 실험안과 공유용 브리프로 바꾼다. 그리고 **검수자**가 모든 산출물을 검증한다. 이들을 묶는 것은 워커가 아니라 상태 머신 — **오케스트레이터** — 다. 오케스트레이터는 워커를 순서대로 구동하고, 검수자가 실패를 반환하면 전체가 아니라 문제가 된 워커 하나만 되돌려 재실행한다.

한 가지 오해를 미리 막아둘 필요가 있다. 워커는 들어오는 질문마다 새로 만들어지지 않는다. 네 명은 고정이다. 입력이 정하는 것은 둘뿐이다 — 어느 워커까지 돌릴 것인가(실행 범위), 그리고 워커에 무엇을 넘길 것인가(파라미터). "분석가를 특화한다"가 아니라 "분석가에게 분석할 채널을 넘긴다"는 식이다. 이 구분이 워커 수가 무한정 늘어나 계약이 흔들리는 사태를 막는다.

이 구조가 주는 것은 추적 가능성이다. 각 단계가 독립된 산출물과 트레이스를 남기므로, 심사위원이 "이 결론이 어떻게 나왔나"를 물으면 단계별로 펼쳐 보일 수 있다. 부분 백트래킹도 가능해진다. 대가는 복잡성이다. 상태 머신을 직접 관리해야 하고, 한 번의 실행에 여러 번의 LLM 호출이 일어나 지연과 비용이 늘어난다. 단일 호출의 단순함을, 들여다볼 수 있고 고쳐 쓸 수 있는 구조와 맞바꾼 셈이다.

> 버린 길: 단일 LangChain류 에이전트(추론을 추적할 수 없고, 환각을 거를 지점도, 부분 재시도도 없음), 케이스마다 특화 에이전트를 동적 생성(워커가 폭증하고 계약이 흔들림).

## 매니지드 플랫폼 대신 직접 쥔 키

이 상태 머신을 무엇으로 돌릴지도 선택이었다. Google Cloud Agent Builder 같은 매니지드 플랫폼은 편의를 준다. 그러나 "검수자 실패 → 특정 워커만 재실행" 같은 도메인 특화 제어 흐름과, 워커가 아무 인덱스에나 손대지 못하도록 검색 도구를 도메인 안전 래퍼로 감싸는 요구를 세밀하게 통제하기 어려웠다.

그래서 매니지드 플랫폼 대신 Google ADK로 직접 오케스트레이션하기로 했다. 워커는 원시 검색 도구를 직접 부르지 않는다. 인덱스가 제한되고 결과가 정규화된 도메인 래퍼 — 콘텐츠 검색, 지표 베이스라인 집계, 팀 메모 검색, 과거 브리프 로드 — 만 호출한다. 상태 머신과 백트래킹 루프는 Python 오케스트레이터가 직접 소유한다.

얻는 것은 완전한 제어권이다. 워커 구동 순서, 되돌림, 컨텍스트 주입을 마음대로 다루고, 모든 단계에 트레이싱을 붙일 수 있다. 포기하는 것은 매니지드 플랫폼이 알아서 해주던 편의이며, 그 자리를 상태 머신 유지보수라는 지속적인 비용이 채운다.

> 버린 길: Agent Builder(도메인 특화 백트래킹·주입 제어가 제한적이고 도구 게이팅이 어려움), 원시 검색 도구를 워커에 직접 노출(근거 무결성과 안전성이 무너짐).

## 검증의 최종 권한은 사람이 쓴 코드에 있다

사용자가 AI 결과를 믿지 못하는 것 — 이것이 기존 도구의 핵심 통증이다. 그래서 검수자(Reviewer Gate)는 이 시스템에서 신뢰가 응축되는 지점이다. 그런데 검증을 어떻게 할 것인가가 까다로웠다. 만약 검증마저 LLM에게 맡기면, 같은 입력에 다른 판정이 나오는 비결정성에 빠지고, 환각으로 환각을 검증하는 역설이 생긴다.

그래서 검수자를 두 층으로 두되, **결정적 코드가 최종 권한을 갖도록** 했다. 첫째 층은 Python의 결정적 검증이다. 스키마 적합성, 그리고 모든 근거 참조(evidence_ref)가 실제로 검색 도구가 반환한 것인지, 존재하지 않는 신호·가설 ID를 가리키지 않는지, 각 실험이 성공 기준과 일정을 갖췄는지를 기계적으로 확인한다. 이 층이 통과/실패를 확정한다. 둘째 층은 Gemini의 보조 비평이다. 무엇이 왜 부족한지 설명을 보태지만, 결정적 층의 실패 판정을 뒤집을 수는 없다. 모든 산출물은 이 결정적 검증을 통과한 뒤에야 사용자 눈앞에 도달한다.

이 선택은 순수한 LLM 판단의 유연함을 포기한다. 규칙으로 명시한 것만 잡히기 때문이다. 그래서 한계도 솔직하다 — 스키마와 규칙으로 표현되지 않은 의미적 약점, 이를테면 그럴듯하지만 근거가 빈약한 가설은 결정적 층이 놓친다. 보조 비평이 이를 보강하지만 차단할 힘은 없다. 그럼에도 이 구조를 택한 이유는 분명하다. "검증을 통과한 것만 보여준다"는 약속은, 그 검증이 재현 가능할 때만 약속이 된다.

> 버린 길: LLM 단독 심판(비결정적이고, 환각으로 환각을 검증하며, 재현 불가), 검증 없이 바로 노출(신뢰라는 핵심 가치를 정면으로 부정).

## 실수에도 종류가 있다

검수자가 실패를 반환할 때, 그 실패의 성격은 둘로 갈린다. 하나는 형식의 실수다 — 필드가 빠졌거나, 타입이 틀렸거나, 명명 규칙을 어겼다. 이건 추론의 오류가 아니라 표현의 실수다. 다른 하나는 의미의 오류다 — 근거를 환각했거나 가설이 빈약하다. 이건 워커가 다시 생각해야만 고쳐진다.

모든 실패를 LLM 재실행으로 처리하는 것은 낭비다. 형식 실수 하나에 비싼 LLM 호출을 쓰고, 결과는 또 비결정적이다. 그래서 둘을 갈라 다뤘다. 형식 오류는 Python의 정규화 코드가 결정적으로 고친다 — LLM을 부르지 않는다. 의미 오류만 해당 워커로 되돌려 다시 추론하게 한다. 애초에 구조화된 출력을 쓰기 때문에 형식 오류 자체가 거의 발생하지 않는다는 점도 이 선택을 뒷받침한다.

원칙은 "결정론을 먼저"다. 고칠 수 있는 것을 추론에 떠넘기지 않는다. 이 분리가 무한 재시도 루프를 막고 교정을 빠르고 재현 가능하게 만든다. 다만 의미 오류의 되돌림이 정해진 재시도 한도를 넘으면, 시스템은 솔직하게 실패를 선언한다.

> 버린 길: 모든 실패를 LLM 재실행으로 통일(형식 실수에 비용을 낭비하고 무한 루프 위험), 정규화마저 LLM에 맡김(결정성을 깨고 환각 가능성을 도로 들임).
