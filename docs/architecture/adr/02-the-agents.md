# 에이전트 결정

에이전트의 분할, 실행 런타임, 검증 권한, 실패 처리에 관한 네 결정.

---

## ADR-0004 · 단일 LLM 대신 4워커 멀티 에이전트 + 오케스트레이터

**상태:** 채택

**맥락:**
해커톤 평가 기준은 에이전트성 증명이다. 단일 LLM이 신호·가설·실험안을 한 번에 생성하면 추론 경로를 단계별로 추적할 수 없고, 환각이나 근거 없는 결론을 거를 중간 지점이 없으며, 한 단계의 오류가 전체 재실행을 강제한다. 제품 루프(신호 → 가설 → 실험)는 단계가 뚜렷해 역할 분리가 가능하다.

**결정:**
역할이 분리된 4개 ADK 워커와 1개 오케스트레이터로 구성한다. 분석가(정량 신호), 전략가(인과 가설), 작가(실험 설계·문서화), 검수자(검증)가 각자 좁은 책임과 draft 스키마를 가진다. Central Orchestrator가 상태 머신으로 워커를 순차 구동하고, 검수자가 실패를 반환하면 해당 워커만 백트래킹시켜 재실행한다. 워커는 질문마다 새로 생성하지 않는다. 입력이 정하는 것은 실행 범위(어느 워커까지)와 파라미터(워커에 넘길 옵션)뿐이다.

**결과:**
각 단계가 독립 산출물과 트레이스를 남겨 추론 경로가 추적 가능하다(ADR-0009, ADR-0010). 검수자 실패 시 전체가 아닌 해당 워커만 부분 백트래킹한다. 책임 경계가 명확해 병렬 개발이 가능하다. 단일 호출의 단순함과 낮은 지연을 포기한다. 상태 머신을 직접 유지해야 한다(ADR-0005). 실행 1회에 다수 LLM 호출이 발생해 지연과 비용이 증가한다.

**대안:**
- 단일 LangChain류 에이전트: 추론 추적, 환각 차단 게이트, 부분 재시도가 모두 불가능하다.
- 케이스별 특화 에이전트 동적 생성: 워커 수가 폭증하고 계약이 흔들린다.

## ADR-0005 · Google ADK 직접 오케스트레이션 (Agent Builder 미사용)

**상태:** 채택

**맥락:**
4워커 백트래킹 상태 머신(ADR-0004)을 어떤 런타임으로 실행할지 결정해야 했다. 매니지드 플랫폼(Agent Builder)은 "검수자 실패 → 특정 워커 재실행" 같은 도메인 특화 제어와, 검색 도구를 도메인 안전 래퍼로 감싸는 요구를 세밀하게 통제하기 어렵다.

**결정:**
Agent Builder를 사용하지 않고 Google ADK로 직접 오케스트레이션한다. 워커는 원시 MCP 도구를 직접 호출하지 않고, 인덱스가 제한되고 결과가 정규화된 도메인 래퍼(`search_content_posts`, `query_metric_baseline`, `search_team_notes`, `load_growth_brief_context`)만 호출한다. 상태 머신과 백트래킹 루프는 Python 오케스트레이터가 소유한다.

**결과:**
워커 구동 순서, 백트래킹, 컨텍스트 주입을 완전히 제어한다. 래퍼가 원시 도구를 가려 근거 무결성과 안전성을 확보한다. 모든 단계에 트레이싱을 부착할 수 있다. 매니지드 플랫폼의 편의를 포기하고 상태 머신 유지보수 비용을 부담한다.

**대안:**
- Google Cloud Agent Builder: 도메인 특화 백트래킹·주입 제어가 제한적이고 도구 게이팅이 어렵다.
- 원시 MCP 도구를 워커에 직접 노출: 임의 인덱스·쿼리 접근으로 근거 무결성과 안전성이 무너진다.

## ADR-0006 · 결정적 검수가 최종 권한, LLM 비평은 보조

**상태:** 채택

**맥락:**
사용자가 AI 결과를 신뢰하지 못하는 것이 기존 도구의 핵심 문제다. 검수 대상에는 검증 가능한 오류(환각된 evidence_ref, 존재하지 않는 ID 참조, 누락된 success_criteria)와 LLM 판단이 필요한 의미적 약점이 섞여 있다. 검증을 LLM에만 맡기면 같은 입력에 다른 판정이 나오는 비결정성과, 환각으로 환각을 검증하는 모순이 생긴다.

**결정:**
검수자를 두 계층으로 두되 결정적 검증이 최종 권한을 가진다. 결정적 계층은 Python에서 Pydantic/JSON Schema와 evidence_ref 집합을 검사하며 pass/fail을 확정한다. Gemini 보조 비평은 설명과 교정 지시를 생성하지만 결정적 실패를 뒤집지 못한다. 모든 산출물은 결정적 검증 통과 후에만 사용자에게 노출된다.

**결과:**
환각 참조, 깨진 ID 참조, 스키마 위반을 결정론적으로 차단한다. "검증 후에만 노출"이라는 신뢰가 재현 가능해진다. 이슈 코드(`UNKNOWN_EVIDENCE_REF`, `MISSING_SUCCESS_CRITERIA` 등)로 정확한 백트래킹 대상을 지정한다(ADR-0007). 순수 LLM 판단의 유연성을 포기한다. 스키마·규칙으로 표현되지 않은 의미적 약점은 결정적 계층이 잡지 못하며, 보조 비평이 보강하나 차단력은 없다.

**대안:**
- LLM-as-judge 단독: 비결정적이고, 환각으로 환각을 검증하며, 재현이 불가능하다.
- 검증 없이 노출: 신뢰라는 핵심 가치에 정면으로 위배된다.

## ADR-0007 · 형식 오류는 결정적 정규화, 의미 오류는 워커 백트래킹

**상태:** 채택

**맥락:**
검수자(ADR-0006)가 반환하는 실패는 두 종류다. 형식 오류(`SCHEMA_INVALID` 등 필드 누락·타입 오류·명명 규칙 위반)는 추론 오류가 아닌 표현 실수다. 의미 오류(환각된 근거, 빈약한 가설)는 워커의 재추론으로만 교정된다. 모든 실패를 LLM 재실행으로 처리하면 형식 실수에 비싼 호출을 낭비하고 결과가 비결정적이다.

**결정:**
오류를 성격에 따라 분리 처리한다. 형식 오류는 Python 정규화 코드로 결정론적으로 교정하며 LLM을 호출하지 않는다. 의미 오류만 해당 워커(주로 전략가/작가)로 백트래킹시켜 재추론한다. 구조화 출력을 사용하므로 형식 오류 자체가 거의 발생하지 않는다.

**결과:**
교정 가능한 것을 추론에 떠넘기지 않는다(결정론 우선). 무한 LLM 재시도 루프를 회피한다. 형식 교정이 빠르고 재현 가능하다. 의미 오류의 백트래킹이 재시도 한도를 초과하면 Python이 Java에 `FAILED`를 반환한다.

**대안:**
- 모든 실패를 LLM 재실행으로 통일: 형식 실수에 비용을 낭비하고 무한 루프 위험이 있다.
- 정규화도 LLM에 위임: 결정성을 깨고 환각 가능성을 재도입한다.
