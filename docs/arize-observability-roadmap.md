# Arize(Phoenix) 관측 고도화 로드맵

LaunchPilot 에이전트의 Arize Phoenix 관측(Observability)을 어디까지 만들었고,
다음에 무엇을 하면 실질적인 효과가 있는지 정리한 문서입니다.
"보기에만 좋은 것(허수)"과 "실제로 품질·효율을 바꾸는 것"을 구분하는 것이 목적입니다.

---

## 1. 한 장 요약

> 에이전트의 모든 행동을 Phoenix에 **기록**(0)하고, 그 기록으로
> **거짓말(할루시네이션)을 차단**(1)하고, 품질 점수를 **진짜 점수로**(2) 만들고,
> **같은 실수를 반복하지 않게**(2.5) 만드는 것까지 완료.
> 남은 것은 점수를 Phoenix UI에 노출하는 것(3)과 운영 규모가 생긴 뒤의 인프라(4·5).

| 단계 | 한 줄 설명 | 효과 | 상태 |
|---|---|---|---|
| 0 | 모든 LLM 호출·도구 호출·검수 결과를 Phoenix에 기록 | 문제 발생 시 원인 추적 가능 | **완료** |
| 1 | 숫자 근거 검증 게이트 (grounding gate) | LLM이 지어낸 숫자를 **차단** | **완료** |
| 2 | 심판(judge)에게 증거 원문 제공 | 품질 점수가 "진짜 점수"가 됨 | **완료** |
| 2.5 | 재시도 피드백 + 프롬프트·온도 보강 | 같은 실수 반복 감소 = 재시도 비용 절감 | **완료** |
| 3 | 점수를 Phoenix evaluation으로 등록 | 대시보드에서 품질 추이 확인 | 다음 후보 |
| 4 | 과거 실패 패턴을 워커 프롬프트에 주입 (reflection) | backtrack 횟수 감소 | 우선순위 하락 (2.5가 대체) |
| 5 | Datasets/Experiments, 비용 대시보드 | 장기 운영용 인프라 | 현 규모에선 보류 |

---

## 2. 현재 상태 (단계 0, 이미 구현됨)

에이전트가 하는 모든 일이 Phoenix Cloud에 트레이스로 남습니다.

```
launchpilot.thread (AGENT)          ← 사용자 턴 전체
└─ launchpilot.orchestrator (CHAIN) ← 분석 파이프라인
   ├─ analyst LLM 호출 + 도구 호출      (ADK 자동 계측)
   │  └─ RETRIEVER 스팬                ← Elastic에서 가져온 증거 문서 기록
   ├─ strategist LLM 호출
   ├─ writer LLM 호출
   └─ launchpilot.reviewer_gate (GUARDRAIL) ← 결정론 검수 결과
      └─ launchpilot.validation (EVALUATOR) ← 통과/실패 + 이슈 코드
```

- 구현 위치: `apps/agent/app/observability.py` (수집 시작), `app/tracing/spans.py` (도메인 스팬)
- 평가 하네스: `app/eval/run_eval.py` — 시나리오 6개를 돌려 결정론 지표 + LLM 심판 점수를 산출
- 검수자(reviewer): ID 정합성, 필수 필드, 캐비앗 누락을 결정론적으로 차단

**이 단계의 가치**: "에이전트가 왜 이런 답을 냈지?"라는 질문에 클릭 몇 번으로 답할 수 있음.
단, 기록은 기록일 뿐 — 품질을 자동으로 올려주지는 않음. 그래서 다음 단계가 필요.

## 3. 단계 1: 숫자 근거 검증 게이트 — 구현 완료

### 무엇을 했나

분석가(analyst)가 보고하는 수치(예: lift_ratio 1.97)가
**실제로 도구가 반환한 값과 일치하는지** 검수자가 대조합니다.

- 증거 도구의 실행 결과를 run 동안 기록 (`app/tools/evidence.py`의 `GroundingCapture`)
- 검수자에 `UNGROUNDED_EVIDENCE` 이슈 코드 추가 (계약 3곳 동기화: schema.json / README / schemas.py)
- 도구가 반환한 적 없는 증거 ref를 인용하거나, 수치가 도구 결과와 다르면(상대 오차 2% 초과)
  검수 실패 → 기존 backtrack 라우팅으로 analyst 재실행
- 반올림(1.966 → 1.97)은 통과, 조작(1.97 → 2.5)은 차단
- 도구 기록이 비어 있으면 검사를 건너뛰므로 기존 동작이 깨지지 않음

### 왜 필요했나 (있던 구멍)

기존 검수자는 "signal이 증거 ref를 **가리키는가**"만 봤습니다.
ref가 가리키는 **실제 값과 일치하는가**는 아무도 안 봤습니다.
즉 LLM이 숫자를 잘못 옮겨 적거나 지어내도 그대로 통과해서 사용자에게 전달됐습니다.

이 시스템의 할루시네이션 위험 표면 전체와 방어 현황:

| 위험 | 방어 |
|---|---|
| 존재하지 않는 signal/hypothesis ID 인용 | 검수자가 차단 |
| 낮은 확신인데 단정적 주장 | 캐비앗 강제 규칙이 차단 |
| 실행 불가능한 실험 항목 | 필수 필드 검사가 차단 |
| 도구 결과와 다른 숫자 보고 | **grounding 게이트가 차단 (이번 구현)** |
| 도구가 반환한 적 없는 증거 ref 인용 | **grounding 게이트가 차단 (이번 구현)** |

### 효용

- **기술**: 결정론적 검사이므로 오탐 없음, LLM 추가 호출 없음(비용 0), 차단권 있음.
  judge(심판)는 점수만 매기고 못 막지만 이 게이트는 실제로 막음.
- **비즈니스**: 사용자가 보는 분석 리포트의 숫자는 전부 데이터에서 검증된 값이라고 보증 가능.
  "AI가 지어낸 수치로 마케팅 의사결정을 했다"는 최악의 시나리오를 구조적으로 제거.
- **검증**: `tests/test_grounding.py` 11개 테스트 — 가짜 ref·조작 수치는 잡히고,
  정상 데이터·반올림은 통과하는 것까지 확인됨.

## 4. 단계 2: 심판(judge)에게 증거 원문 제공 — 구현 완료

### 무엇을 했나

LLM 심판(`app/eval/judge.py`)의 입력에 최종 결과물(payload)뿐 아니라
**도구가 실제로 반환한 값(TOOL EVIDENCE)**을 함께 넣고,
"도구 증거를 진실의 원천으로 삼아 교차검증하라"는 지시를 추가했습니다.
평가 러너(`app/eval/run_eval.py`)가 턴 전체를 capture로 감싸 같은 ground truth를
결정론 지표와 심판 양쪽에 전달합니다.

### 왜 필요했나

기존 심판은 결과물만 보고 채점했습니다. "증거가 있어 보이는가"는 판단해도
"증거와 일치하는가"는 물리적으로 판단할 수 없었습니다.
즉 기존 signal_validity 점수는 **그럴듯함 점수**였지 근거 점수가 아니었습니다.
그 상태로 점수를 쌓으면 지표 자체가 허수가 됩니다.

### 효용

- **기술**: 단계 1이 "숫자 일치"라는 결정론 검사라면, 이것은 "해석이 증거를 과장하지 않았는가"라는
  의미론적 검사. 둘이 합쳐져 grounding 검증이 완성됨.
- **비즈니스**: 품질 리포트의 점수를 외부(심사위원, 고객)에 제시할 때 "근거 대조 기반 채점"이라고 말할 수 있음.

## 4.5. 단계 2.5: 재시도 피드백 + 프롬프트·온도 보강 — 구현 완료

게이트가 잡은 다음 "재시도가 같은 실수를 반복하지 않게" 만드는 예방 묶음 3개.

### 무엇을 했나

1. **재시도 피드백 루프** (`app/orchestrator.py` + `app/agents/workers.py`):
   기존에는 검수 실패 후 재실행할 때 **처음과 똑같은 프롬프트**로 다시 시켰습니다.
   LLM은 자기가 왜 틀렸는지 모르니 같은 실수를 반복하기 쉬웠습니다.
   이제 검수자가 만드는 `retry_instruction`(실패 이유 + 수정 지시)을
   근본 원인 워커의 재실행 프롬프트에 주입합니다.
2. **숫자 전사 규칙** (`app/agents/instructions.py`):
   analyst 지시문에 "current_value/baseline_value/lift_ratio는 도구가 반환한 값을
   그대로 복사하라, 재계산·반올림·추정 금지"를 명시. 게이트(사후 차단)의 사전 예방 짝.
3. **temperature 하향** (`app/agents/adk_agents.py`):
   숫자 옮겨적기·분류 작업에 샘플링 다양성은 전사 오류만 늘립니다.
   analyst·router는 0.0, strategist·writer는 0.2로 고정 (chat은 기본값 유지).

### 효용

- **기술**: backtrack 1회 = LLM 호출 1~3회 추가. 피드백 주입으로 재시도 성공률이 오르면
  그 비용이 직접 줄어듦. 전사 규칙·저온도는 backtrack 자체를 줄임.
- **비즈니스**: 응답 시간 단축 + 토큰 비용 절감. 게이트(차단)와 합쳐
  "잡고, 알려주고, 고치게 한다"는 자기 교정 루프가 완성됨.
- **검증**: `tests/test_feedback.py` — 피드백이 프롬프트에 붙는 것, 스텁 경로가 깨지지 않는 것 확인.

## 5. 단계 3: 점수를 Phoenix evaluation으로 등록

### 무엇을 하나

지금 judge 점수와 검수 통과 여부는 스팬 속성(attribute)으로만 존재합니다.
이를 Phoenix의 1급 **evaluation/annotation**으로 등록하면 UI에서 직접 다룰 수 있게 됩니다.

- judge 3축 점수 + reviewer pass/fail을 해당 trace의 스팬에 evaluation으로 부착
- 구현 시 arize-phoenix-client의 evaluation API 버전 확인 필요 (버전별 API 차이 있음)

### 효용

- **기술**: "hypothesis_grounding 3점 미만인 run만 보기" 같은 필터링, 시계열 품질 추이 집계가 UI에서 가능.
  reflection(단계 4)이 raw 텍스트 파싱 대신 구조화된 평가 데이터를 읽을 수 있게 됨.
- **비즈니스**: 데모/심사에서 "품질이 측정되고 추적되는 시스템"을 화면으로 보여줄 수 있음.
- **솔직한 판정**: 점수를 보고 프롬프트를 고치는 반복 루프를 실제로 돌릴 때만 유의미.
  한 번 만들고 안 보면 허수. 단계 2 이후에 할 것 (점수가 믿을 만해진 다음).

## 6. 단계 4: 실패 패턴 프롬프트 주입 (reflection 루프 닫기)

### 무엇을 하나

현재 `app/agents/reflection.py`는 Phoenix에서 과거 검수 실패를 읽어 요약까지 만들지만,
그 요약이 **로그에만 찍히고 버려집니다** (`orchestrator.py`의 reflection 블록 참고).
이 요약을 analyst/strategist의 instruction에 컨텍스트로 주입해 루프를 닫습니다.

예: "과거 LOW_CONFIDENCE_WITHOUT_CAVEAT 실패 12회 → 낮은 확신의 가설에는 반드시 캐비앗을 달 것"

### 효용

- **기술**: 같은 검수 실패의 반복이 줄어 backtrack 횟수 감소 = LLM 재호출 비용·지연 감소.
  backtrack_count가 이미 스팬에 기록되므로 효과를 전후 비교로 측정 가능.
- **비즈니스**: 응답 시간 단축, 토큰 비용 절감.

### 솔직한 주의사항 — 단계 2.5 이후 우선순위 하락

골든패스는 도메인·워커가 고정이라, 실패 패턴이 일정하다면
**instruction을 정적으로 고치는 것이 더 싸고 효과가 같습니다.**
단계 2.5에서 그 정적 수정(전사 규칙)과 턴 내 피드백 루프(retry_instruction 주입)를
이미 적용했습니다. reflection이 추가로 기여할 수 있는 것은 "**턴을 넘어** 반복되는
실패 패턴"뿐인데, 그런 패턴이 실제로 관측될 때만 가치가 있습니다.

권장: Phoenix에서 backtrack_count·이슈 코드 추이를 보다가 같은 이슈가 여러 턴에 걸쳐
반복되면 그때 켤 것. 동적 주입은 "스스로 학습하는 에이전트" 데모 스토리로는
가치가 있으나, 그것은 별개의 판단 축.

## 7. 단계 5: 보류 항목 (현 규모에서 허수 가능성 높음)

| 항목 | 내용 | 보류 이유 |
|---|---|---|
| Phoenix Datasets/Experiments | 시나리오를 Phoenix에 등록, 실험 단위 비교 | 시나리오 6개·해커톤 기간. 로컬 MD 리포트로 충분. 반복 실험 수십 회 규모에서 가치 발생 |
| 비용/지연 대시보드 | turn당 토큰·지연, 이슈 코드별 backtrack 빈도 모니터 | 파이프라인 고정·트래픽 낮음 → 비용 구조 이미 자명. 토큰 수는 자동 계측이 이미 수집 중 |
| 승인/거절 human annotation | Java 측 approve/reject를 trace에 라벨로 부착 | 진짜 ground-truth 루프이나 Java 측 작업 + 실사용자 데이터 필요. 시기상조 |

버리는 것이 아니라 **트래픽과 반복 개선 주기가 생기면 그때 켜는 것**입니다.

## 8. 진행 현황과 이후 추천

```
[완료] 0단계  Phoenix 기록 (스팬 5종 + 자동 계측)
[완료] 1단계  grounding 게이트          ← 할루시네이션 차단
[완료] 2단계  judge에 증거 제공          ← 품질 점수를 진짜 점수로
[완료] 2.5단계 재시도 피드백 + 전사 규칙 + 저온도 ← 같은 실수 반복 방지
   ↓
[다음] 3단계  evaluation 등록           ← 믿을 만해진 점수를 Phoenix UI에 노출
[옵션] 수치 결정론 주입                  ← 아래 참고
[관망] 4단계  reflection 주입           ← 턴을 넘는 반복 실패가 관측되면
[보류] 5단계  인프라 항목                ← 운영 규모가 생기면
```

핵심 논리: **차단(1) 없이 측정(2·3)만 하면 보기 좋은 숫자일 뿐이고,
측정 없이 개선만 하면 효과를 증명할 수 없다.** 차단·측정·교정(2.5)까지 끝났으니
남은 것은 노출(3)과, 규모가 생긴 뒤의 인프라(4·5)다.

### 이후 추천사항 상세

**1순위 — 단계 3 (evaluation 등록)**: 점수가 근거 대조 기반으로 믿을 만해졌으므로
이제 Phoenix UI에 올릴 가치가 생겼음. 데모에서 "품질이 측정·추적되는 시스템"을
화면으로 보여줄 수 있는 가장 싼 방법. 구현 전 arize-phoenix-client의
evaluation API 버전 확인 필요.

**옵션 — 수치 결정론 주입 (효과 최강, 변경 중간)**: 현재는 LLM이 도구 숫자를
옮겨 적고 게이트가 틀리면 잡아서 재실행(감지 + 재시도 비용). 대안은 analyst가
"어떤 metric/channel이 signal인지 + 증거 ref"만 반환하고, **수치 필드는 Python이
GroundingCapture에서 직접 채우는 것**. 숫자 할루시네이션이 "잡는 대상"에서
