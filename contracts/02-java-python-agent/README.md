# LaunchPilot Java-Python Agent Internal Contract

Status: Draft v0.1  
Boundary: Java Spring Boot Business Backend <-> Python FastAPI Agent Service  
Last updated: 2026-06-01

## Purpose

This contract defines the internal HTTP boundary between the Java Backend and the Python AI Agent component.

Java owns:

- public frontend API,
- `agent_run_id` generation,
- Tomcat thread isolation through its async job manager,
- CSV parsing and Elastic ingestion,
- final human approval and append-only persistence to `calendar_events` and `growth_briefs`.

Python owns:

- agent state machine execution,
- Gemini calls,
- Elastic MCP evidence retrieval,
- Phoenix MCP self-introspection,
- OpenTelemetry / OpenInference tracing,
- structured candidate payload generation.

The Python service does not process final human approval. It generates candidate signals, hypotheses, and experiment plans only.

## Base Conventions

Base URL from Java to Python: configurable, recommended `http://agent-service:8000`

Internal path prefix: `/internal/agent`

Content type: `application/json`

Authentication: omitted for hackathon v0.1. If needed later, use a service-to-service header such as `X-Internal-Token`.

Timestamps: ISO 8601 strings with timezone offsets.

IDs:

- `agent_run_id`: generated by Java, format `run_*`.
- `trace_context.request_id`: generated by Java per public request, format `req_*`.
- Python must echo `agent_run_id` in every response.

## Runtime Invariants

- Java calls Python once to start a run, then polls Python for status.
- Python must return quickly from `POST /internal/agent/runs`; long-running work happens in a background task.
- Python status values must be compatible with the frontend-facing Java status enum.
- Python `payload` must be structurally compatible with `contracts/01-frontend-java/openapi.yaml` `AgentResultPayload`.
- Python may include additional internal diagnostics under `agent_diagnostics`; Java may log this but should not expose all diagnostics to the frontend by default.
- `WAITING_FOR_APPROVAL` is the Python terminal state for successful candidate generation.
- `SUCCESS` is reserved for the Java approval lifecycle and should not normally be emitted by Python v0.1.

## Status Mapping

| Python status | Java public status | Meaning |
| --- | --- | --- |
| `PENDING` | `PENDING` | Run accepted but worker has not started. |
| `RUNNING_SIGNAL_DETECTION` | `RUNNING_SIGNAL_DETECTION` | Data Analyst worker is extracting quantitative signals. |
| `RUNNING_EVIDENCE_SEARCH` | `RUNNING_EVIDENCE_SEARCH` | Elastic/Phoenix retrieval tools are collecting evidence. |
| `RUNNING_HYPOTHESIS_GENERATION` | `RUNNING_HYPOTHESIS_GENERATION` | Strategist worker is creating causal hypotheses. |
| `RUNNING_EXPERIMENT_GENERATION` | `RUNNING_EXPERIMENT_GENERATION` | Writer/Strategist path is creating experiment candidates. |
| `WAITING_FOR_APPROVAL` | `WAITING_FOR_APPROVAL` | Candidate payload passed validation and is ready for UI review. |
| `FAILED` | `FAILED` | Agent run failed after retries or unrecoverable infrastructure error. |

## 1. Start Internal Agent Run

`POST /internal/agent/runs`

Purpose: Java asks Python to start an agent run using a Java-generated run ID.

Request:

```json
{
  "agent_run_id": "run_20260601_001",
  "workspace_id": "demo_workspace",
  "campaign_id": "camp_comeback_teaser",
  "question": "What should we test next week?",
  "date_range": {
    "start": "2026-05-25",
    "end": "2026-06-01"
  },
  "parent_brief_id": null,
  "trace_context": {
    "request_id": "req_20260601_001",
    "source": "java-backend",
    "otel_trace_id": null
  }
}
```

Response: `202 Accepted`

```json
{
  "ok": true,
  "agent_run_id": "run_20260601_001",
  "status": "PENDING",
  "accepted_at": "2026-06-01T16:31:00+09:00"
}
```

Idempotency rule:

- If Java retries the same `agent_run_id`, Python should return the current run status instead of starting a duplicate run.
- If the same `agent_run_id` is reused with a different request body, Python should return `409 Conflict`.

## 3. Cancel Internal Agent Run

`POST /internal/agent/runs/{agent_run_id}/cancel`

Purpose: optional operational endpoint for Java to stop a run when the frontend session is abandoned or an operator aborts work.

Response: `202 Accepted`

```json
{
  "ok": true,
  "agent_run_id": "run_20260601_001",
  "status": "FAILED",
  "cancelled_at": "2026-06-01T16:31:08+09:00"
}
```

For v0.1, cancellation may be best-effort. Java should treat a late `WAITING_FOR_APPROVAL` result after cancellation as stale if Java already discarded the run.

## Error Shape

All internal errors use the same envelope:

```json
{
  "ok": false,
  "error": {
    "code": "RUN_ID_CONFLICT",
    "message": "agent_run_id already exists with a different request body.",
    "details": {
      "agent_run_id": "run_20260601_001"
    },
    "request_id": "req_20260601_001"
  }
}
```

Recommended error codes:

- `INVALID_REQUEST`
- `RUN_NOT_FOUND`
- `RUN_ID_CONFLICT`
- `AGENT_BUSY`
- `TOOL_CALL_FAILED`
- `GEMINI_FAILED`
- `VALIDATION_FAILED`
- `INTERNAL_AGENT_ERROR`

## Java Mapping Rules

Java should expose only these fields to the frontend:

- `agent_run_id`
- `status`
- `current_stage`
- `retry_count`
- `error_message`
- `payload`
- `tool_call_logs`

Java should keep these fields internal by default:

- `agent_diagnostics`
- `started_at`
- `updated_at`
- `completed_at`
- raw Python failure stack or provider error bodies
